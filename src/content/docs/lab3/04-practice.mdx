### 根因分析

 `sys_yield` 或是时钟中断的链路如下：

```text
sys_yield() / SupervisorTimer
  → suspend_current_and_run_next()
      → mark_current_suspended()   ← 【目前这里啥也没做】
      → run_next_task()
            → find_next_task()
                  → 寻找 TaskStatus::Ready 状态的候选人！
```

`find_next_task` 会在就绪队列中轮转寻找标记为 `TaskStatus::Ready` 的任务。

 `mark_current_suspended` 函数并没有把当前任务自己的状态从 `Running` 切换回 `Ready`。所以当它调用 `yield()` 试图休息时，调度器在扫描全场时，发现除了还没开始运行的就绪任务外，根本没有将其视为合法的“等待中任务”。调度器继续寻找，阴差阳错这会让未完成就绪状态重塑的 A 不断重新占据 CPU 主导判断权。
