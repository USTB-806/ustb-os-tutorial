## 为什么需要地址空间？

在前面的实验中，所有用户程序被预先加载到物理内存的固定位置（
`APP_BASE_ADDRESS + i * APP_SIZE_LIMIT`），存在以下根本性缺陷：

| 问题 | 说明 |
|------|------|
| **隔离性差** | 程序可以直接读写其他程序或内核的内存地址 |
| **大小受限** | 每个程序的大小上限固定为 `APP_SIZE_LIMIT` |
| **地址固定** | 程序必须在链接时知道自己的绝对物理地址，无法复用 |
| **碎片化** | 随应用增删，物理内存出现难以利用的碎片 |

**地址空间（Address Space）** 通过在每个程序和物理内存之间引入一层**虚拟地址到物理地址的映射**，解决上述所有问题：每个程序都"以为"自己独占整个虚拟地址空间，

## RISC-V SV39 分页方案

本章采用 RISC-V 的 **SV39** 分页模式（Sv39 = Supervisor mode, 39-bit virtual）：

### 地址位宽

```
63          39 38                          12 11          0
┌─────────────┬──────────────────────────────┬─────────────┐
│  符号扩展位  │        虚拟页号 VPN[27:0]     │  页内偏移   │
│   (25位)    │   VPN[2]  VPN[1]  VPN[0]     │   (12位)    |
|             |    9位     9位    9位        |             |
└─────────────┴──────────────────────────────┴─────────────┘
                                      

55                                          12 11          0
┌────────────────────────────────────────────┬─────────────┐
│            物理页号 PPN[43:0]               │  页内偏移   │
│                   (44位)                    │   (12位)   │
└────────────────────────────────────────────┴─────────────┘
```

<Aside type="note">
虽然虚拟地址是 64 位，但在 SV39 模式下只有低 39 位有效，其余高位必须是第 38 位的符号扩展（全 0 或全 1），
否则 CPU 会触发异常。
</Aside>

### 页的大小

 **4 KiB = 4096 字节 = 2^12 字节**，因此偏移占 12 位。

```rust
//kernel/src/config.rs
pub const PAGE_SIZE: usize = 0x1000;       // 4096
pub const PAGE_SIZE_BITS: usize = 0xc;    // 12
```

### 三级页表结构

SV39 使用**三级页表**，虚拟页号被分成三段，每段 9 位，作为各级页表的索引：

```
       VPN = [ VPN[2] | VPN[1] | VPN[0] ]
                  9位       9位       9位
  satp.root_ppn
       │
       ▼
  一级页表（512项）
  [VPN[2]] → PPN_1
                 │
                 ▼
            二级页表（512项）
            [VPN[1]] → PPN_2
                           │
                           ▼
                      三级页表（512项）
                      [VPN[0]] → 目标 PPN
                                       │
                                       ▼
                                  物理地址 = PPN << 12 | 页内偏移
```

每一级页表刚好占用一个物理页（512项 × 8字节 = 4096字节），
 512 × 512 × 512 × 4KiB = **256 GiB** 的虚拟地址空间。

## satp 寄存器：开启分页

CPU 通过 `satp` CSR（Supervisor Address Translation and Protection）控制分页：

```
satp[63:60] = 8    → 启用 SV39 分页模式
satp[43:0]  = PPN  → 根页表的物理页号
```

 Rust 代码中对应：

```rust
pub fn token(&self) -> usize {
    8usize << 60 | self.root_ppn.0
}

pub fn activate(&self) {
    let satp = self.page_table.token();
    unsafe {
        satp::write(satp);
        asm!("sfence.vma");  // 刷新 TLB
    }
}
```

<Aside type="caution">
 `satp` 后，必须立即执行 `sfence.vma` 指令刷新 TLB（地址转换缓存），
 CPU 可能仍使用旧的缓存映射，导致错误的地址翻译。
</Aside>

## 本章地址空间布局

### 用户地址空间

```
0xFFFF_FFFF_FFFF_F000  ┌──────────────────┐ TRAMPOLINE（跳板页，只读执行）
0xFFFF_FFFF_FFFF_E000  ├──────────────────┤ TrapContext（陷阱上下文）
                       │                  │
                       │   (unmapped)     │
                       │                  │
user_stack_top         ├──────────────────┤ 用户栈
user_stack_bottom      ├──────────────────┤ 保护页（guard page，不映射）
                       ├──────────────────┤ 程序数据段
                       ├──────────────────┤ 程序只读段
0x0000_0000_0000_0000  ├──────────────────┤ 程序代码段（ELF 入口点）
```

:::note
注意这并不是一个合理的内存地址空间布局（至少没有heap空间），但是我们目前的os不追求太全面，而是为了让大家理解**虚拟地址空间**。
:::


- `TRAMPOLINE = usize::MAX - PAGE_SIZE + 1`，因此位于最高虚拟页（`0xFFFF_FFFF_FFFF_F000`）。
- `TRAP_CONTEXT_BASE = TRAMPOLINE - PAGE_SIZE`，因此 TrapContext 紧贴在 Trampoline 下方一页（`0xFFFF_FFFF_FFFF_E000`）。
- 用户栈来自 `from_elf` 的 `user_stack_bottom..user_stack_top` 区间，并显式预留了一页 guard page（不映射）用于防止栈越界静默破坏内存。

### 内核地址空间

```
0xFFFF_FFFF_FFFF_F000  ┌──────────────────┐ Trampoline（与物理地址对齐）
                       │    MMIO 寄存器    │ 恒等映射
                       │    物理内存       │ 恒等映射
                       │    .bss/.data     │ 恒等映射
                       │    .rodata        │ 恒等映射
                       │    .text          │ 恒等映射（只读执行）
```

<Aside type="tip">
**跳板页（Trampoline）** 是内核和所有用户程序地址空间中都映射到同一物理地址的特殊页。
当 Trap 从用户态切换到内核态时，CPU 会切换页表，但这不影响
`__alltraps` 和 `__restore` 代码。

更具体地说：
1. 进入用户态前，内核把 `stvec` 设为 `TRAMPOLINE`，因此一旦用户态发生异常/中断，硬件总是先跳到这页中的 `__alltraps`。
2. `__alltraps` 会先保存寄存器，再切换到内核页表与内核栈，随后进入 Rust 的 `trap_handler`。
3. 返回用户态时，`trap_return` 计算 `__restore` 在 Trampoline 页内的虚拟地址并跳转执行，最后 `sret` 回到用户程序。

之所以必须“内核页表和用户页表都映射同一跳板页”，是为了保证页表切换前后，Trap 入口/出口代码始终可取指执行，不会在最关键的过渡瞬间丢失执行流。
</Aside>
